Introduction to Project
This is a minimalist rental management webapp for business owner. No login needed, one click on your unique URL and you are in. Mobile responsiveness is key, as almost all my clients use phone to open this webapp.
- My clients are small business owners, they purchase camerabody and lense (both are expensive) and rent then out to fans attending concert, where fans use these device to take photos and videos of their favorite performer on the stage. Rental is daily rate, so fans have no need to spend huge on these camerabody and lense, this is my client's business model.
- The calculation of revenue and cost is related to rental order, but not related to device calendar. Device calendar is often different from rental order, as there are different type of fulfillment, currently there is local order and remote order, and for a same rental period, for example 3 days, local fulfillment occupies less days that remote fulfillment.
- User can CRUD concert event and rental order. device calendar is updated automatically and accordingly.
- a global fixed header that shows the app name, alway fixed on top of page during scrolling, click to go to main page. 
- a global footer that attached to end of page, with copyright info.
- a navigation bar at bottom when view on mobile, first botton is Calendar, second is Orders, third is Inventory. 
- Calendar page is main page, it is view-only. 
    - calendar page display a big calendar with info from device_calendar table, user can easily see when and which devices are occupied for which concert. 
- Orders page has two view, Create and Past Orders. Create can create Concert, or Rental Order. 
    - read the Tables section below in this doc to build accordingly.
    - Create and Past Order should be in 2 different files/endpoints, for modularity purpose, as we may change layout in the future. 
    - create concert and create rental order should be in 2 different files/endpoints, for modularity purpose, as we may change layout in the future. 
    - to build the create concert function, refer to Tables section below in this doc.
    - to build the create rental order function, refer to Tables section below in this doc.
    - in past order, list all past order, and display 3 metrics on the top of the list, total net income(revenue - shipping fee - intern fee), total revenue(revenue = daily rate * number of days), total fees (shipping fee + intern fee)

- Inventory lists all the camerabody and lenses, and allow user to CRUD camerabody and lenses.

SQL Database
- we use a SQLite db in the backend, and because the webapp will be deployed on Cloudflare Worker, we will use Cloudflare D1 as db.
- only ones marked optional are optional.

SQL Database Tables:
- rental_orders
    order_id (PK) (generated by db, not by user)
    concert_id (FK) (an order must link to a concert)
    start_date (date)
    end_date (date)
    camerabody_id (FK)
    lense_id (FK)
    daily_rate (money)
    shipping_fee (optional, default to 0)
    intern_fee (optional, default to 0)
    notes (optional, text, default to null or whatever is a best practice here)
- concert_calendar
    concert_id (PK)  (generated by db, not by user)
    title (text)
    location_type (enum, two options, local or remote)
    start_date (date)
    end_date (date)
- device_calendar
    order_id (PK, FK)  (generated by db, not by user)
    camerabody_id (FK)
    lense_id (FK)
    start_date (date)
    end_date (date)
- device_asset
    device_id (PK)  (generated by db, not by user)
    device_name (text)
    device_type (enum, two options, camerabody or lense)

Backend: 
- in server, (not in db), we need to check the date range of rental orders must sit inside corresponding concert's date range. or to overlap on start and end.
- in server, (not in db), for rental_order and device_calendar queries respectively, we need to check that either or both of camerabody_id and lense_id must be present, which means the two cannot be none at the same time.
